ESP32-S3 + APC UPS (USB HID) — Progress Summary (Current: v8 SUCCESS)
Date: 2026-03-05 (America/New_York)

Purpose
- Build a small ESP32-S3 “NUT node” that connects to an APC UPS over USB OTG (HID) and exposes UPS status over LAN.
- Current focus: prove reliable USB host enumeration + HID interrupt IN decoding; then build a clean status/state model; then add network services.

Hardware (confirmed stable)
- ESP32-S3 board with USB OTG used as host to APC UPS.
- Second USB port provides power; OTG VBUS powered via soldered USB->OTG wiring on rear of board.
- Hardware/cables/soldering unchanged across tests; earlier tests confirmed UPS transitions (load → on-battery), so hardware is not the issue.

Environment
- ESP-IDF v5.3.1-dirty (Windows host, idf.py monitor on COM3).
- Target UPS: APC UPS VID:PID 051d:0002.
- Observed USB: FULL speed; EP0 MPS 64; HID interrupt IN endpoint 0x81 MPS 8.

What is Working Now (v8 — SUCCESS)
1) USB host lifecycle is reliable
- usb_host_install OK
- usb_host_client_register OK
- NEW_DEV events received; device opens correctly; DEV_GONE handled.
- Re-plugging UPS works repeatedly; state resets on attach and clears on detach.

2) Descriptor parsing (from active config descriptor) is reliable
- Active config wTotalLength=34
- HID interface located: intf=0 alt=0 subclass=0x00 proto=0x00
- HID report descriptor length parsed from HID descriptor (0x21): 1049 bytes
- Interrupt IN EP located: 0x81, MPS=8
- Interface claim succeeds.

3) Interrupt IN streaming (change-only logging) is stable
- Reports arrive repeatedly with minimal log spam (only on change).
- Observed repeating report IDs: 0x06, 0x0C, 0x13, 0x14, 0x16, 0x21.

4) Decoding implemented (Milestone 4) — verified by logs
- Report ID 0x0C: battery charge + runtime (little-endian runtime seconds)
  Example: batt=98%, runtime=35280 s
- Report ID 0x13: AC present flag
  Example: AC_present=1
- Report ID 0x16: 32-bit flags (little-endian)
  Example: flags=0x0000000D

5) State + NUT-ish status logs (Milestone 5) — verified by logs
- State resets on attach (“Milestone 5: state reset on attach.”)
- State clears on detach (“Milestone 5: state cleared on detach.”)
- A “NUT-ish status” block prints when key decoded fields arrive/refresh:
  - battery.charge
  - battery.runtime
  - input.utility.present (derived from AC_present when known)
  - ups.flags (when known)
  - ups.status: currently conservative:
    - OL when AC_present=1
    - UNKNOWN before AC report arrives

Key Evidence from v8 Monitor Logs
- NEW_DEV → Connected VID:PID=051d:0002 → parse config → claim interface
- Interrupt IN reports:
  - 0x0C decoded to batt=98% runtime=35280s
  - 0x13 decoded to AC_present=1 → ups.status becomes OL
  - 0x16 decoded to flags=0x0000000D
- DEV_GONE received and handled; subsequent re-attach repeats success.

Known Issue / De-prioritized (v7 only)
- v7 attempted a control GET_DESCRIPTOR(report) dump and got:
  - control submit failed: ESP_ERR_INVALID_ARG
- This does NOT affect interrupt IN reading and is not required for progress (descriptor length is already parsed from config 0x21).
- v8 avoids this path and is stable.
