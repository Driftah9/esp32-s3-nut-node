1) MASTER REQUIREMENTS (USER CONFIRMED)
============================================================
A) Core Function
1. The ESP32-S3 shall act as a NUT node (upsd-compatible server) that a NUT client can query over tcp/3493.
2. The firmware shall scan USB devices plugged into the OTG port and:
   - Identify HID-class UPS-like devices by interface descriptors / HID descriptor / ability to fetch a Report Descriptor.
   - Attempt to pull data for any compatible UPS (no fixed VID/PID requirement).
   - Handle USB detach/reattach cleanly (state reset on attach, close endpoints on detach).
3. The firmware shall decode and expose UPS status values over NUT variables; minimum currently implemented:
   - battery.charge
   - battery.runtime
   - input.utility.present
   - ups.flags
   - ups.status (e.g., OL)
   - (Extend over time toward NUT-style completeness.)

B) Networking / Provisioning
4. The firmware shall provide a SoftAP mode used for configuration:
   - Default IP: 192.168.4.1 (captive portal recommended).
   - SoftAP password must be changeable via the web interface.
5. The firmware shall provide a web interface accessible in SoftAP mode that allows:
   - Joining a local Wi‑Fi network (STA mode) by entering SSID + password.
   - Switching operation to LAN mode (STA) and obtaining IP from DHCP server on the local LAN
     (example network: 10.0.0.0/16).
   - Persisting network settings across reboot.
6. After joining local Wi‑Fi, the firmware shall run the NUT TCP server on the LAN IP so NUT clients on the LAN can connect.

C) Identity / Access Control
7. The firmware shall allow changing the UPS identifier used by NUT clients:
   - Allow configuring the NUT UPS name (e.g., ups) and how clients reference it as device@IP_address.
   - Expose configured identifier in LIST UPS and related outputs.
8. The firmware shall support NUT authentication configuration:
   - Allow changing username/password used by NUT authentication (USER/PASS sequence).
   - Store credentials securely (as feasible on ESP32; see Security notes).

D) Web UI Behavior
9. Any settings page that modifies persistent configuration shall provide:
   - Save/apply action
   - A "Reboot device" option/button after save (and/or auto-reboot option)

============================================================
2) RECOMMENDED / LIKELY MISSING REQUIREMENTS (TO HELP FINISH THE PROJECT)
============================================================
These are not explicitly requested, but are commonly needed to complete a robust “NUT node appliance”:

R1. Persistence & Factory Reset
- Store config (Wi‑Fi creds, SoftAP password, NUT creds, UPS name, mode flags) in NVS.
- Provide a factory-reset mechanism:
  - Web UI button + confirmation
  - Optional physical long-press (BOOT button) to wipe NVS

R2. Mode & Fallback Policy
- Define boot policy when STA connection fails:
  - e.g., try STA for N seconds → fall back to SoftAP+config portal.
- Option to run SoftAP+STA concurrently (APSTA) for recovery while on LAN.

R3. Discoverability
- mDNS hostname (e.g., upsd-esp32.local) so you don’t need to remember the IP.
- Optional: show LAN IP in /status and config portal.

R4. Captive Portal UX
- DNS redirect + HTTP redirect so opening any site while connected to SoftAP lands on the config UI.

R5. NUT Protocol Coverage / Interop
- Implement enough upsd command set for real NUT clients:
  - HELP (optional but nice)
  - USER/PASS + permissions behavior
  - LIST CMD, INSTCMD (if you plan control commands)
  - LIST RW / SET VAR (if you want remote changes)
- Validate with standard NUT tools:
  - upsc, upscmd, upsmon, nut-scanner (where applicable)

R6. UPS Data Completeness
- Expand HID decode mapping:
  - input.voltage, input.frequency (if device reports)
  - battery.voltage, battery.temperature (if device reports)
  - ups.load, output.voltage, beeper status, alarm status
- Keep “unknown RID tracking” logging behind a debug flag to avoid noisy logs.

R7. Multi-UPS Behavior
- Decide policy: only one UPS at a time (simpler) vs multiple devices.
- If one-at-a-time: reject additional UPS connections with clear log message.

R8. Security Posture (LAN)
- Decide whether to allow plaintext NUT credentials on LAN (upsd typically is plaintext unless wrapped/secured).
- If you want stronger security:
  - Place behind trusted LAN only; optionally allow enabling/disabling auth.
  - Consider TLS termination externally (router/reverse proxy/VPN) rather than on-device.
- Rate limit login attempts to avoid trivial brute-force.

R9. OTA & Maintainability (Optional but practical)
- Add OTA update endpoint (web UI upload) or HTTPS OTA.
- Add a “build/version” string in UI and in NUT vars (driver.version, device.serial, etc.).

R10. Time & Logging
- If runtime/ages are shown, consider time sync (SNTP) on LAN.
- Remote log view in web UI (optional) for debugging.
