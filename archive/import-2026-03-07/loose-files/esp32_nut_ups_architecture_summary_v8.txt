UPDATED CONTEXT & STATUS (v8 – note Option A flash failure / Option B success, updated 2026-03-05)

---------------------------------------------------------------------
1) Project Goal
---------------------------------------------------------------------
Build ESP32-S3 “UPS nodes” that:

• Connect to APC UPS units that expose USB HID via the RJ50 communication port
• Read UPS telemetry using the ESP32-S3 USB OTG host interface
• Expose a NUT-compatible TCP service so standard NUT clients can query the node
• Provide a web UI for configuration and monitoring
• Operate as distributed UPS monitoring nodes across a LAN

Primary use case:
Centralize monitoring of multiple UPS units without needing a full computer
attached to each UPS.

Milestone 3 – HID report monitor (change-only) + capture raw frames
---------------------------------------------------------------
COMPLETE (validated 2026-03-04)

What we achieved:

- Claimed HID interface 0, alt 0 on APC UPS VID:PID 051d:0002
- Found interrupt IN endpoint EP=0x81, MPS=8
- Started a continuous interrupt-IN reader that prints reports **only when the payload changes**
  (keeps logs readable and avoids “polling spam”)

Observed raw report IDs / frames (examples):

- 06 00 00 08
- 0C 64 A0 8C
- 13 01
- 14 00 00
- 16 0C 00 00 00
- 21 06

Device detach / reattach behavior (validated 2026-03-05):

- On unplug, USB host reports “Device gone”, the in-flight INT-IN transfer completes with status=7,
  and resubmitting returns ESP_ERR_INVALID_STATE.
- Our logic then receives DEV_GONE and stops the session cleanly.
- On re-plug, NEW_DEV arrives and the session re-opens/claims again successfully.

Key log lines from the validated run:

- “HID Report Descriptor length = 1049 bytes (from HID descriptor 0x21)”
- “Starting interrupt IN reader: EP=0x81 MPS=8 (change-only)”
- “HID IN changed (...) ...” (repeats only when frames differ)

Notes / gotchas:

- A prior Milestone 3 attempt that *waited synchronously* for the HID Report Descriptor could time out and then crash.
  For Milestone 3 we do not block the USB client task; we move forward with streaming interrupt-IN reports.

Next (Milestone 3.1):

- Build a parser that maps these raw HID report IDs/fields into NUT-style values
  (battery.charge, battery.runtime, input.voltage, ups.status, etc.), and expose them internally as a struct.

Milestone 4: Start decoding key UPS reports (basic)
----------------------------------------------
STATUS: COMPLETE (basic decode proven on APC 051d:0002)

What we decode now (based on observed report IDs in interrupt-IN stream):
- Report ID 0x0C (4 bytes payload observed): battery percent + runtime seconds
  - Example: 0C 64 A0 8C  -> batt=100% runtime=36000 s
  - Example: 0C 63 A0 8C  -> batt= 99% runtime=36000 s
  - Example: 0C 63 38 8B  -> batt= 99% runtime=35640 s
  - Example: 0C 62 D0 89  -> batt= 98% runtime=35280 s

- Report ID 0x13 (2 bytes payload observed): AC present boolean
  - Example: 13 01 -> AC_present=1
  - Example: 13 00 -> AC_present=0

- Report ID 0x16 (5 bytes payload observed): status/flags bitfield (little-endian, 32-bit)
  - Example: 16 0C 00 00 00 -> flags=0x0000000C
  - Example: 16 0A 00 00 00 -> flags=0x0000000A
  - Example: 16 0D 00 00 00 -> flags=0x0000000D
  - Current code logs a bit-diff view on change (which bits flipped). Meanings are still TBD.

Power-fail / restore test results (APC load very light, so small changes expected):
- AC pulled ~20 seconds:
  - 0x13 changed 1 -> 0
  - 0x16 changed 0x0000000C -> 0x0000000A (diff 0x00000006)
  - 0x0C battery dropped 100% -> 99% shortly after

- AC restored:
  - 0x13 changed 0 -> 1
  - 0x0C runtime and battery updated (99%->98% seen during recovery)
  - 0x16 changed 0x0000000A -> 0x0000000D (diff 0x00000007)

Notes / gotchas:
- One attempted build failed with: usb_transfer_t has no member named 'setup'
  - Keep using the known-good control-transfer pattern for report-descriptor fetch on ESP-IDF v5.3.1.
- On one attach, log showed “HID Report Descriptor received: 1057 bytes” and extra trailing bytes printed after the expected descriptor.
  - Treat this as a bug/edge case in our dump path (length mismatch / buffer overrun risk) to fix next.

---------------------------------------------------------------------

The HID report descriptor length captured from the UPS is:

1049 bytes

The descriptor contains multiple report IDs.

Observed report IDs include:

06
0C
13
14
16
21

These IDs correspond to different telemetry groups inside the UPS.

Typical fields expected from APC HID UPS devices:

• battery charge percentage
• battery runtime remaining
• input voltage
• output voltage
• load percentage
• UPS status flags
• alarm states

The descriptor will be parsed to determine:

• Report ID structure
• Field bit lengths
• Scaling factors
• Unit conversions

Once decoded, these will be mapped to NUT variables.
